# Встраивание в RuSIEM

## Основной принцип

Модуль должен работать внутри RuSIEM или в тесно связанном контуре RuSIEM.
Основной источник данных - уже нормализованный поток событий самой системы.
Внешние REST, webhook и syslog нужны только для разработки, тестирования или
совместимости, но не являются целевой промышленной схемой.

## Входящие данные из RuSIEM

Целевой контракт прототипа:

- `POST /api/v1/rusiem/internal/events`

В промышленной реализации этот endpoint может быть заменен внутренним вызовом,
очередью или подпиской на поток нормализованных событий RuSIEM.

Модуль получает:

- нормализованные поля события;
- tenant/node-контекст;
- связанные симптомы;
- идентификатор правила корреляции, если он уже есть;
- контекст актива;
- контекст пользователя;
- системные справочники.

## Возврат результата в SIEM

Для промышленного контура результат должен возвращаться не как внешний алерт, а
как родной контекст RuSIEM:

- поля прогноза в событие;
- новый симптом с весом риска;
- обогащение существующего инцидента;
- создание кандидата в инцидент при высоком `risk_score`;
- запись объяснений и рекомендаций для аналитика.

## Рекомендуемые поля для события-прогноза

- `prediction.id`
- `prediction.risk_score`
- `prediction.probability`
- `prediction.risk_level`
- `prediction.threat_class`
- `prediction.confidence`
- `prediction.explanations`
- `prediction.recommendations`
- `source.event_id`
- `source.ip`
- `destination.ip`
- `user.name`
- `asset.id`

## Режим нормализованных событий

Для RuSIEM предпочтителен режим `POST /api/v1/rusiem/internal/events`. В этом
случае модуль не выполняет повторную нормализацию и сразу запускает:

1. обогащение;
2. анализ последовательности;
3. risk scoring;
4. формирование объяснений и рекомендаций.

Минимальный контракт:

```json
{
  "event_id": "rusiem-internal-1",
  "event_time": "2026-06-11T10:00:00Z",
  "event_type": "auth_failed",
  "src_ip": "203.0.113.10",
  "dst_ip": "10.0.0.5",
  "username": "ivanov",
  "asset_id": "srv-ad-01",
  "severity": "high",
  "message": "Failed login",
  "fields": {
    "event.category": "authentication",
    "event.outcome": "failure"
  },
  "context": {
    "tenant_id": "tenant-1",
    "node_id": "node-main",
    "symptom_ids": ["symptom-auth-failed"],
    "asset": {
      "criticality": "high"
    }
  }
}
```

## Dev/test-режимы

В прототипе оставлены дополнительные входы:

- `POST /api/v1/events`;
- `POST /api/v1/events/normalized`;
- `POST /api/v1/webhook/rusiem`;
- UDP syslog `5514`.

Они нужны для автономной проверки алгоритмов и демонстрации, но целевая
архитектура должна опираться на внутренние сервисы RuSIEM.

## Связь с документацией RuSIEM

Документация RuSIEM указывает основные каналы и порты приема событий:

- agent: `443/3515 TCP`;
- syslog: `514`, `5014`, `5013 TCP/UDP`;
- netflow: `9999`;
- SNMP: `162 UDP`.

Для модуля прогноза удобнее не вмешиваться в стандартный pipeline RuSIEM, а
работать как внешний enrichment/scoring-сервис: получать копию событий, считать
прогноз и возвращать результат как отдельное событие или контекст для инцидента.
