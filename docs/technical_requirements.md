# Техническое задание: внутренний модуль прогноза инцидентов ИБ для RuSIEM

## Назначение

Модуль предназначен для работы внутри RuSIEM и прогнозирования вероятности
инцидентов информационной безопасности на основе нормализованного потока событий
самой системы, системных справочников, активов, пользователей, симптомов,
корреляционных правил и внешних контекстных источников.

## Цели

- Выявлять признаки подготовки или развития инцидента.
- Рассчитывать вероятность угрозы и уровень риска.
- Формировать рекомендации по реагированию.
- Возвращать результат в RuSIEM как родной контекст события, симптома или
  инцидента.

## Функциональные требования

1. Прием событий:
   - Основной режим: прием уже нормализованных событий из RuSIEM.
   - Получение tenant/node-контекста.
   - Получение связанных симптомов, правил корреляции, пользователя и актива.
   - Dev/test-режим: REST API, webhook и UDP syslog для автономной проверки.

2. Нормализация:
   - В целевом режиме повторная нормализация не выполняется.
   - Модуль принимает нормализованные поля RuSIEM.
   - Dev/test-режим может приводить сырые события к внутреннему контракту.

3. Обогащение:
   - GeoIP: страна, ASN, признак публичного адреса.
   - WHOIS: регистратор, возраст домена.
   - Threat Intelligence: признаки known bad IP/domain/hash/url.
   - Контекст актива: критичность, владелец, бизнес-система.

4. Аналитика последовательностей:
   - Подсчет событий в скользящем временном окне.
   - Выявление паттернов: failed auth burst, success after failures, web attack burst.
   - Подготовка признаков для скоринга или ML-модели.

5. Risk scoring:
   - Расчет `probability` в диапазоне `0..1`.
   - Расчет `risk_score` в диапазоне `0..100`.
   - Классификация уровня риска: `low`, `medium`, `high`, `critical`.
   - Классификация класса угрозы.

6. Объяснимость:
   - Возврат списка факторов, повлиявших на score.
   - Возврат рекомендаций для аналитика.
   - Сохранение нормализованного события и обогащения в ответе.

## Нефункциональные требования

- Горизонтальное масштабирование ingestion-слоя.
- Идемпотентность обработки по `event_id`.
- Наблюдаемость: метрики EPS, latency, error rate, распределение risk score.
- Безопасность: TLS, токены/API keys, аудит обращений.
- Отказоустойчивость: очередь между ingestion и scoring.
- Возможность отключения внешних обогащений при недоступности провайдера.
